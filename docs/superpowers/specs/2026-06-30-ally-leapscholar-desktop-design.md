# LeapScholar Ally — Desktop Web App Design

Date: 2026-06-30

## Background

`leapscholar_figma_export (1).html` is a static Figma export containing 11 mobile screens and 11 matching desktop (1280px) screens for an "Ally" AI-mentor study-abroad funnel: Landing → Welcome → 7 questions → Loading → Results. This spec covers rebuilding the **desktop view only** as a real, interactive Next.js application, with the final Results screen powered by a live OpenAI call instead of the export's hardcoded mock data.

Mobile layouts in the export are out of scope for this pass.

## Goals

- Pixel-accurate recreation of the desktop visual design (colors, type, spacing, components) from the Figma export.
- A genuinely interactive flow: users click through the 7 questions, drag the budget slider, reorder priorities, and reach a Results screen.
- Results are generated by calling OpenAI with the user's collected answers, returned as structured JSON, and rendered into the same `uni-card` / locked-row UI as the mock.
- No backend persistence, no real lead-capture submission — this pass is the UI + AI matching only.

## Non-Goals

- Mobile/responsive layouts (tracked as a future follow-up; the export's mobile markup can inform that work later).
- Real lead-capture form wired to a backend ("Unlock all 10" stays a styled, non-functional button).
- User accounts, persistence across sessions, analytics.
- A curated university dataset — the AI generates match data from its own knowledge (see Trade-offs).

## Architecture

Next.js 14+, App Router, TypeScript. Single route (`/`) renders a client component (`page.tsx`) that drives all 11 screens as an internal state machine — no client-side routing between screens, no page reloads.

```
app/
  layout.tsx            - loads Plus Jakarta Sans font, imports globals.css
  page.tsx               - "use client"; owns `step` and `answers` state, renders the active screen
  api/match/route.ts     - server-side Route Handler, calls OpenAI, returns MatchResult[]
components/
  screens/
    Landing.tsx
    Welcome.tsx
    Question.tsx          - generic option-card screen, reused for Q1 (qualification), Q2 (IELTS),
                             Q3 (field), Q4 (countries, multi-select), Q7 (timeline)
    BudgetQuestion.tsx     - Q5, slider
    PriorityQuestion.tsx   - Q6, reorderable ranked list
    Loading.tsx            - animated step list + fetch to /api/match
    Results.tsx            - renders MatchResult[] into top-2 cards + 10 locked rows
  ui/
    Nav.tsx, PreviewCard.tsx, StatStrip.tsx, FeatureRow.tsx, TrustStrip.tsx, Reviews.tsx
       - landing-page sub-sections, split out for readability
lib/
  questions.ts            - config for Q1-Q4 & Q7: { id, prompt, options: [{icon/flag, label, sub}] }
  types.ts                - Answers, MatchResult, Step types shared client/server
styles/
  globals.css             - ported desktop CSS from the export (custom properties, .nav, .hero-desktop,
                             .option-card, .uni-card, etc.) Mobile-only and export-wrapper rules
                             (.canvas, .phone-shell, .browser-shell, .hero-mobile, ...) are dropped.
.env.local.example         - OPENAI_API_KEY=, OPENAI_MODEL=gpt-4o-mini
```

## State & Flow

```ts
type Step =
  | "landing" | "welcome"
  | "q1" | "q2" | "q3" | "q4" | "q5" | "q6" | "q7"
  | "loading" | "results";

type Answers = {
  qualification?: string;
  ielts?: string;
  field?: string;
  countries: string[];
  budget: number;        // lakhs INR, default 25, range 5-50
  priorities: string[];  // ordered, most-important first
  timeline?: string;
};
```

Per-question interaction matches the affordances visible in the export:

- **Q1, Q2, Q3, Q7** — single-select option-cards. Clicking an option records the answer and immediately auto-advances to the next step (no separate "Continue" button, matching the mock).
- **Q4 (countries)** — multi-select toggle cards. A "Continue →" button is disabled until at least one country is selected, then advances.
- **Q5 (budget)** — drag slider, ₹5L–₹50L+ in steps of 5. The "At this budget…" suggestion text updates live as the slider moves. "Confirm budget →" advances.
- **Q6 (priorities)** — ranked list of 3 items with up/down reorder controls (replacing the mock's static drag-handle glyph with functional buttons, since drag-and-drop isn't part of the visual spec). "Confirm ranking →" advances.
- Every question screen's "← Back" steps to the previous step, preserving all collected answers (no data loss on back navigation).
- "Tell Ally directly" (the custom option on Q1, Q2, Q4, Q7) reveals an inline text input in place of the option grid; submitting non-empty text records it as that question's answer and advances.

On entering `loading`, `page.tsx` POSTs the full `Answers` object to `/api/match`. The Loading screen plays the four-step animation from the mock (each `loading-step` lights up in sequence on a timer) and only transitions to `results` once the fetch resolves AND a minimum 2.5s has elapsed (avoids a jarring flash on fast responses). If the fetch fails, the Loading screen shows an inline error message with a "Try again" button that retries the same POST — no silent fallback to fake data.

## OpenAI Integration

`app/api/match/route.ts`:

- Accepts `POST { answers: Answers }`.
- Builds a prompt summarizing the user's profile (qualification, English test status, field, target countries, budget, ranked priorities, intended start).
- Calls OpenAI Chat Completions with a structured-output JSON schema enforcing:

```ts
type MatchResult = {
  university: string;
  country: string;
  course: string;
  matchPercent: number;       // 0-100
  tuitionPerYearINR: number;  // raw rupee amount, formatted client-side
  placementPercent: number;   // 0-100
  qsRank: number;
  reason: string;             // 1-2 sentence "why Ally picked this"
};
// API response body: { matches: MatchResult[] }  -- exactly 12 items, sorted matchPercent desc
```

- Model defaults to `gpt-4o-mini`, overridable via `OPENAI_MODEL` env var.
- The route returns `{ matches }` on success or `{ error: string }` with a non-200 status on failure (missing key, OpenAI error, malformed response).

`Results.tsx` renders `matches[0]` and `matches[1]` as full `uni-card`s with the "✦ Best match" / "2nd best" badges (computed from index, not hardcoded), and `matches[2..11]` as the blurred, locked `full-row` list — same visual treatment as the export, but the underlying 10 entries are real model output rather than the hardcoded Melbourne/Manchester/etc. rows.

## Error Handling

- Missing/invalid `OPENAI_API_KEY`: route handler returns a clear error message; Loading screen surfaces it with a retry button.
- OpenAI returns malformed/incomplete JSON: treated as a failure (structured-output schema should prevent this in practice, but the route validates `matches.length === 12` before responding 200).
- Network failure on the client fetch: same retry-button UX as above.

## Trade-offs / Open Questions Resolved During Brainstorming

- **Fully AI-generated vs curated dataset:** chose fully AI-generated (no seed dataset) per explicit decision — faster to build, accepting that tuition/rank/placement numbers are model estimates and may vary slightly between runs rather than being authoritative.
- **Desktop-only scope:** mobile layouts from the export are not built in this pass; revisit as a separate spec if/when needed.
- **"Unlock all 10" button:** stays non-functional in this pass — no lead-capture form or backend submission is in scope.

## Testing

- Manual verification via `npm run dev`: click through the full funnel (Landing → Welcome → Q1-Q7 → Loading → Results) confirming visual match to the desktop export and correct state transitions (back button, multi-select toggle, slider drag, priority reorder, custom text input).
- Verify `/api/match` independently with a sample `Answers` payload (e.g. via curl) to confirm the structured response shape before wiring the UI fetch.
- No automated test suite is in scope for this pass given the project's small, UI-focused surface area.
